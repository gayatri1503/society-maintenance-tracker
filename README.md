# Society Maintenance Tracker

A full-stack platform for apartment societies to manage maintenance complaints, track their full lifecycle with an audit trail, run a notice board, and keep residents informed via email.

## Tech Stack

**Backend:** FastAPI, PostgreSQL (Neon), SQLAlchemy, Alembic, JWT auth (python-jose), bcrypt
**Frontend:** React (Vite), React Router, Axios
**Photo storage:** Cloudinary
**Email:** Resend

## Setup Guide

### Prerequisites
- Python 3.12+
- Node.js 18+
- A PostgreSQL database (this project uses [Neon](https://neon.tech), free tier)
- A [Cloudinary](https://cloudinary.com) account (free tier)
- A [Resend](https://resend.com) account (free tier)

### Backend

```bash
cd backend
python -m venv venv
venv\Scripts\activate        # Windows
# source venv/bin/activate   # Mac/Linux

pip install -r requirements.txt

cp .env.example .env
# Fill in .env with your real values (see below)

alembic upgrade head         # creates all tables

uvicorn app.main:app --reload
```

Backend runs at `http://localhost:8000`. Interactive API docs at `http://localhost:8000/docs`.

### Frontend

```bash
cd frontend
npm install

cp .env.example .env
# Set VITE_API_URL to your backend URL (http://localhost:8000 for local dev)

npm run dev
```

Frontend runs at `http://localhost:5173`.

### Creating an admin account

All accounts created via `/auth/register` are residents by default (by design — admin accounts are not self-service). To create an admin:

1. Register normally through the app or via `POST /auth/register`
2. Promote that account directly in the database:
```sql
UPDATE users SET role = 'admin' WHERE email = 'your-admin-email@example.com';
```

## API Documentation

Full interactive documentation (Swagger UI) is auto-generated by FastAPI at `/docs` when the backend is running. Summary below:

### Auth
| Method | Endpoint | Access | Description |
|---|---|---|---|
| POST | `/auth/register` | Public | Register a new resident account |
| POST | `/auth/login` | Public | Log in, returns JWT access token |
| GET | `/auth/me` | Authenticated | Get current logged-in user's details |

### Complaints
| Method | Endpoint | Access | Description |
|---|---|---|---|
| POST | `/complaints` | Resident | Create a complaint (multipart form: category, description, optional photo) |
| GET | `/complaints/me` | Resident | Get all of the current resident's complaints with full history |
| GET | `/complaints` | Admin | Get all complaints. Query params: `category`, `status`, `date_from`, `date_to`. Overdue complaints surface first |
| PATCH | `/complaints/{id}/status` | Admin | Update status (Open/In Progress/Resolved), appends a history entry. Resolved complaints are locked from further updates |
| PATCH | `/complaints/{id}/priority` | Admin | Update priority (Low/Medium/High) |

### Notices
| Method | Endpoint | Access | Description |
|---|---|---|---|
| POST | `/notices` | Admin | Post a notice, optionally marked important |
| GET | `/notices` | Authenticated | List all notices; important ones pinned to top |

### Dashboard
| Method | Endpoint | Access | Description |
|---|---|---|---|
| GET | `/dashboard` | Admin | Aggregate counts: total complaints, by status, by category, overdue count |

### Authentication

All authenticated endpoints require a header:
Authorization: Bearer <access_token>
Tokens are obtained from `/auth/login` and expire after `ACCESS_TOKEN_EXPIRE_MINUTES` (default 60).

## Database Schema

### `users`
| Column | Type | Notes |
|---|---|---|
| id | Integer | Primary key |
| name | String | |
| email | String | Unique |
| password_hash | String | bcrypt hash |
| role | Enum | `resident` \| `admin` |
| flat_number | String | Nullable |
| created_at | DateTime | |

### `complaints`
| Column | Type | Notes |
|---|---|---|
| id | Integer | Primary key |
| resident_id | Integer | FK → users.id |
| category | String | |
| description | Text | |
| photo_url | String | Nullable, Cloudinary URL |
| current_status | Enum | `Open` \| `In Progress` \| `Resolved` — denormalized cache of latest status |
| priority | Enum | `Low` \| `Medium` \| `High` |
| created_at | DateTime | Used for overdue calculation |

### `complaint_status_history`
| Column | Type | Notes |
|---|---|---|
| id | Integer | Primary key |
| complaint_id | Integer | FK → complaints.id |
| status | Enum | Status at this point in time |
| note | Text | Nullable, optional admin note |
| changed_by | Integer | FK → users.id (actor who made the change) |
| changed_at | DateTime | |

Append-only audit log — every status change (including complaint creation) writes a new row here. Never updated or deleted, only inserted.

### `notices`
| Column | Type | Notes |
|---|---|---|
| id | Integer | Primary key |
| title | String | |
| body | Text | |
| is_important | Boolean | Pins notice to top of list |
| posted_by | Integer | FK → users.id |
| created_at | DateTime | |

### Relationships
- One `User` (resident) → many `Complaint`
- One `Complaint` → many `ComplaintStatusHistory` (chronological, ordered by `changed_at`)
- One `User` → many `ComplaintStatusHistory` entries (as the actor who made each change)
- One `User` (admin) → many `Notice`

## Known Limitations (Free-Tier Constraints)

- **Email delivery:** Resend's free tier sandbox sender (`onboarding@resend.dev`) can only deliver to the email address used to sign up for Resend, unless a custom domain is verified. In production, a verified domain would be used to email all residents freely.
- **Backend cold starts:** If deployed on Render's free tier, the backend spins down after 15 minutes of inactivity and takes 30-60 seconds to wake on the next request.
